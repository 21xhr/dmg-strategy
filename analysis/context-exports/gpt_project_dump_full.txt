# **21-Minute Session Scheduler**
# **21-Minute Session Scheduler**
The autonomous scheduler system executes **one single session tick** exactly 21 minutes after the challenge is manually launched by the Game Master, and then automatically halts execution.

| Component | Responsibility | Frequency / Trigger |
| :---- | :---- | :---- |
| **scheduler.ts** | The main cron runner. Calls the core service logic. | **Every 10 seconds** (\*/10 \* \* \* \* \*) |
| **challengeService.processAutomaticSessionTick** | **Core Logic:** Finds the single challenge where isExecuting: true. | Runs every 10 seconds. |
| **Logic Check** | Calculates the elapsed time since timestampLastSessionTick. |  |
| **Tick Threshold** | **21 minutes** (SESSION\_DURATION\_MS). |  |
| **Session Tick & Stop** | If the threshold is passed: **increments currentSessionCount**, dispatches SESSION\_TICKED event, and **CRITICALLY: sets isExecuting: false and timestampLastSessionTick: null**. | Triggers once, 21 minutes after manual launch. |
| **Completion** | If currentSessionCount equals totalSessions, sets status: COMPLETED. | Triggers on the final successful tick. |
| **Resumption** | **Manual Game Master re-execution** is required to advance to the next session. | Game Master command (/challenges/execute). |

# **Command Submission Security Layers**
This appendix summarizes the defense-in-depth measures protecting the main command ingress path and the related user actions.

| Layer | Component / Mechanism | Type of Protection |
| :---- | :---- | :---- |
| Authentication | authenticateUser middleware | Resolves the external identity, ensures the request contains a valid platform identity context, and attaches the internal DMG user identity before command execution. |
| Resource throttling | User-scoped rate limiting[^1] | Limits repeated command submissions per authenticated user and reduces immediate request spam or resource exhaustion. |
| Financial deterrent (user) | Quadratic cost logic | Increases the NUMBERS cost of repeated high-impact actions such as submission and push, discouraging repetitive abuse. |
| Financial deterrent (system) | Balance checks | Stops impossible or unfunded paid actions before full execution. |
| Token integrity (webform flows) | JWT access token | Prevents protected frontend actions from being called without a valid, short-lived token tied to a verified identity. |
| Client-side pacing | Verification and success-state delays | Adds small time barriers to reduce hyper-fast repeated interaction and improve the readability of protected web flows. |

## **Why this matters**

These layers do not rely on one single gate.  
They combine identity resolution, rate limiting, cost pressure, balance validation, token checks, and frontend pacing to make abuse harder while keeping the product usable.

# **Detailed Removal/Refund Option Matrix**
\!remove can only be executed by the original challenge author.

The command does not have a separate direct cost, but it triggers a refund calculation based on prior push spending on that challenge.

## **Refund calculation basis**

When a challenge is removed, the system calculates a 21% refund share from the push contributions already made on that challenge.

This refund is divided into two parts:

* Author's Share \= 21% of the author's own push contributions on that challenge

* Pushers' Share \= 21% of the external pushers' contributions on that challenge

The selected refund option determines where those two shares are directed.

## **Refund Distribution Options**

| Option | RefundOption | Destination of Funds | Description |
| :---- | :---- | :---- | :---- |
| A | community\_forfeit | 100% to Community Chest | The full calculated 21% refund (Author's Share \+ Pushers' Share) is directed to the Community Chest. This is the default option if no option is specified. |
| B | author\_and\_chest | Author's Share to Author; Pushers' Share to Community Chest | The author reclaims their own 21% share. The external pushers' 21% share is redirected to the Community Chest. |
| C | author\_and\_pushers | Author's Share to Author; Pushers' Share to External Pushers | The author reclaims their own 21% share. The external pushers' 21% share is refunded back to those external pushers. This requires external refund calls. |

## **Reliability / Execution Notes**

* Community Chest redirection happens as part of the removal itself, inside the main atomic database operation.

* Refunds back to the author and/or external pushers happen after the removal has been committed, because they rely on external refund operations.

## **Why this matters**

This system makes challenge removal more than a simple delete action. It creates:

* author accountability

* optional generosity through the Community Chest

* strategic redistribution of previously committed NUMBERS

# Authentication / Command Ingress Flow
## Purpose

This flow documents how an external chat command becomes an authenticated internal DMG action.

## Flow Summary

This flow has four stages:

1. **External command ingress**
   - a connected integration sends a command payload to `/api/v1/command`
   - the payload includes:
     - `command`
     - `user`
     - `platform`
     - optional `args`

2. **Authentication and identity resolution**
   - `authenticateUser` reads the external platform identity from the request body
   - it uses `platformName` + `platformId` to resolve the matching internal DMG user
   - if the identity does not yet exist, it creates a new `User / Account` pair
   - it then attaches a flattened authenticated request context to the request object:
        - `req.userId` as the resolved internal DMG user identity
        - `req.platformId` as a flattened copy of the platform ID from the request body
        - `req.platformName` as a flattened copy of the platform name from the request body

3. **Rate limiting and dispatch**
   - `rateLimitUserCommands` applies user-scoped rate limiting after the internal user identity is known
   - the request then reaches `dispatchCommand()`
   - the dispatcher parses the command arguments and routes the authenticated command to the correct backend service function

4. **Service execution**
   - the target service function runs using the resolved internal user identity together with the platform information copied onto the request object
   - the backend returns a command result for the integration to display or act on

## Sequence Diagram

```mermaid
sequenceDiagram
    actor U as User
    participant I as Chat / Integration
    participant CMD as /api/v1/command
    participant AUTH as authenticateUser
    participant USR as findOrCreateUser
    participant RL as rateLimitUserCommands
    participant DISP as dispatchCommand
    participant SVC as Target Service Function
    participant DB as Prisma / DB

    U->>I: send command
    I->>CMD: POST command payload
    CMD->>AUTH: authenticate request
    AUTH->>USR: resolve or create internal user
    USR-->>AUTH: central userId
    AUTH-->>CMD: attach req.userId / req.platformId / req.platformName
    CMD->>RL: apply user-scoped rate limit
    RL-->>CMD: allow request
    CMD->>DISP: dispatch authenticated command
    DISP->>SVC: call target service function
    SVC->>DB: read / write system state
    DB-->>SVC: result
    SVC-->>DISP: command result
    DISP-->>CMD: formatted response
    CMD-->>I: response payload
```

## Implementation Notes

- the external payload arrives as nested data inside `req.body`
- `authenticateUser` reads the external identity from that payload, resolves the internal DMG user, and then attaches a flattened authenticated request context to the Express request object
- `req.userId` is the resolved internal DMG identity
- `req.platformId` and `req.platformName` are flattened request-level copies of the platform identity used for the current command
- `dispatchCommand()` receives already-authenticated commands; it does not perform identity resolution itself
- first-time users can enter the system through this flow because identity resolution may create a new internal `User / Account` pair during command ingress

# Secure Challenge Submission Flow
## Purpose

This flow documents how a chat command becomes a protected web submission path.

## Flow Summary

This flow has three stages:

1. **Command ingress and link generation**
   - the user triggers `!challengesubmit` through a connected chat integration
   - the command reaches `/api/v1/command`
   - `authenticateUser` resolves or creates the internal DMG user identity
   - the dispatcher routes the command to `processSubmissionLinkGeneration(...)`
   - the backend computes the current submission context and generates a signed time-limited token
   - the user receives a secure submission link in chat

2. **Secure link verification in the browser**
   - the user opens the secure link in the browser
   - the frontend extracts the token from the URL
   - `/token/verify-challengesubmitform` verifies the token
   - the backend checks that `payload.userId` still exists
   - the backend returns the current submission context so the frontend can display the correct identity and cost state before submission

3. **Final submission and coordinated payment flow**
   - the user fills the form and submits the challenge payload together with the token
   - `/user/submit/web` verifies the token again
   - the backend validates the payload and recalculates the current submission context
   - the backend confirms payment through Lumia
   - after successful payment confirmation, the backend commits challenge creation and related database updates
   - the frontend receives the created challenge data and updated user state

# Secure Explorer Access Flow
## Purpose

This flow documents how a chat command becomes a protected Explorer access path, and how entry fee payment is handled as a separate browser-side unlock step.

## Flow Summary

This flow has four stages:

1. **Command ingress and secure link generation**
   - the user triggers `!explorer` through a connected chat integration
   - the command reaches `/api/v1/command`
   - `authenticateUser` resolves or creates the internal DMG user identity
   - the dispatcher routes the command to `processExplorerLinkGeneration(...)`
   - the backend generates a signed secure token and builds the Explorer URL
   - the user receives the secure Explorer link in chat

2. **Secure link verification in the browser**
   - the user opens the secure Explorer link in the browser
   - the frontend extracts the token from the URL or local storage
   - `/token/verify-explorer` verifies the token
   - the backend checks that `payload.userId` still exists
   - the backend returns the current Explorer access state for that internal identity
   - the response includes:
     - resolved identity (`username`, `platformName`)
     - dataset scope (`isDemo`)
     - current price tier (`unitPrice`)
     - merit state (`hasMerit`)
     - whether access is currently active (`accessGranted`)
     - remaining time (`minutesRemaining`, `explorerAccessUntil`)

3. **Paid access grant or extension**
   - if access is not currently active, the frontend shows the Explorer paywall overlay
   - the user selects the number of 21-minute access units to buy
   - the frontend submits `token + units` to `/token/grant-explorer-access`
   - the backend verifies the token again
   - the backend reloads the internal user and matching platform account
   - the backend determines:
     - whether the user qualifies for merit pricing
     - the effective unit price
     - the effective granted units
     - the total minutes granted
     - the total price to deduct
   - the backend confirms payment through Lumia
   - after successful payment confirmation, the backend updates:
     - `user.explorerAccessUntil`
     - `user.totalNumbersSpent`
     - `account.currentBalance`
     - global spending ledger on user `id = 1`
     - user activity timestamps
   - the frontend receives the new access expiry and starts the local countdown

4. **Entitled Explorer data access**
   - once access is active, the frontend loads `/challenges`
   - the frontend polls `/challenges/delta`
   - both routes extract the Explorer token from the `Authorization: Bearer ...` header
   - the backend verifies the token and resolves the entitled dataset scope through `resolveExplorerEntitledScope(...)`
   - the backend checks that `user.explorerAccessUntil > now`
   - if access is valid:
     - demo users receive the demo dataset only
     - non-demo users receive the full dataset
   - if access is missing or expired, the API rejects access and the frontend returns to the paywall state

## Implementation Notes
- command ingress is the point where user/account resolution may create a new internal DMG user
- the Explorer secure link does not grant access by itself; it only transfers identity securely to the browser
- `/token/verify-explorer` does **not** process payment
- `/token/verify-explorer` is the browser bootstrap step that returns identity, merit pricing state, dataset scope, and current access status
- `/token/grant-explorer-access` is the only route that deducts NUMBERS and grants or extends Explorer access
- token verification is performed again during access grant; the payment/grant step does not trust prior frontend verification
- the backend validates that the platform account identified by `payload.platformId + payload.platformName` still belongs to `payload.userId`
- merit pricing is derived from `user.totalNumbersSpent >= EXPLORER_MERIT_THRESHOLD`
- current configuration is:
  - standard price: `21` NUMBERS per `21` minutes
  - merit price: `3` NUMBERS per `21` minutes
  - step size: `21` minutes
- demo users are restricted twice:
  - dataset scope resolves to `demo`
  - granted units are forcibly capped to `1` by `getEffectiveExplorerGrantUnits(...)`
- Explorer collection routes do not trust frontend scope parameters; scope is derived server-side from the verified token
- `/challenges` and `/challenges/delta` require an `Authorization: Bearer <token>` header, not a query param
- `/challenges` and `/challenges/delta` both enforce active entitlement by checking `user.explorerAccessUntil > now`
- if entitlement expires, the backend returns `EXPLORER_ACCESS_REQUIRED`, and the frontend should return to the access overlay
- public challenge detail route `/challenges/:id` remains public and is not gated by Explorer entitlement
- the `!explorer` command generates a secure Explorer link through `processExplorerLinkGeneration(...)`
- the link token is used to transfer identity securely into the browser
- paid Explorer access is controlled by `user.explorerAccessUntil`
- in other words, the Explorer JWT is primarily an identity transport token, while Explorer entitlement lives in database state

# **Quadratic Cost Dual-Logic (Submission vs Push)**
DMG uses two independent quadratic cost formulas based on two distinct user counters.

### **Challenge Submission Cost (**`!challengesubmit`**)**
**Counter**: `Users.daily_submission_count`

**Logic**: per-user, daily-reset

**Purpose**: prevent submission spam and incentivize more thoughtful challenge proposals

**Equation**: 

$210 \times (N+1)^2$

**Equation details:**

* N = the number of challenges the user has already submitted that day

* N + 1 = the sequential number of the next challenge being submitted

* 210 = the base cost multiplier

* $(N + 1)^2$ = the quadratic escalation factor

### **Challenge Promotion Cost (**`!push`**)**

**Counter**: the number of pushes already made by the same user on the same challenge

**Logic**: per-user, per-challenge, never-resets

**Purpose**: prevent one user from dominating promotion on a single challenge and force more strategic distribution of NUMBERS

**Equation**: 

$\sum_{i=1}^{\mathrm{qty}} \left[ 21 \times (\mathrm{Prev} + i)^2 \right]$

**Equation details:**

* qty = the number of pushes requested in the current transaction  
* Prev = the number of pushes the same user has already made on that specific challenge  
* i = the summation index, starting at 1  
* Prev + i = the sequential number of each new push in the requested batch  
* 21 = the base push cost multiplier  
* $(Prev + i)^2$ \= the quadratic escalation factor for each individual push
* $\sum_{i=1}^{\mathrm{qty}}$ = sums the cost of every requested push in the batch

---

## **Quadratic Cost Calculation Example: Per-User Logic**

This example demonstrates the cost calculation for the \`\!push\` command, which uses a per-user, per-challenge, never-resets quadratic model.

### **Scenario Parameters**

* Base Price (Multiplier) \= 21

* Current user push count for this challenge \= 2

* Quantity requested (qty) \= 5

This means the user is requesting their 3rd, 4th, 5th, 6th, and 7th push on the same challenge.

### **Detailed Calculation**

| Push \# | Formula Term | Calculation | Cost (NUMBERS) |
| ----- | :---- | :---- | :---- |
| 1st (user’s 3rd push) | $(2 + 1)^2$ | $3^2$ \= 9 | 21 × 9 \= 189 |
| 2nd (user’s 4th push) | $(2 + 2)^2$ | $4^2$ \= 16 | 21 × 16 \= 336 |
| 3rd (user’s 5th push) | $(2 + 3)^2$ | $5^2$ \= 25 | 21 × 25 \= 525 |
| 4th (user’s 6th push) | $(2 + 4)^2$ | $6^2$ \= 36 | 21 × 36 \= 756 |
| 5th (user’s 7th push) | $(2 + 5)^2$ | $7^2$ \= 49 | 21 × 49 \= 1,029 |
| **Total Cost**: 2,835 NUMBERS |  |  |  |

### **Quote Record Output**

A successful quote would store:

* quoted\_quantity: 5

* quoted\_cost: 2835

# **Recurring Challenge Cadence Enforcement**
This appendix explains how cadence enforcement works for RECURRING challenges.

## **Purpose**

A recurring challenge is not judged only by total completion.  
It is also judged by whether the required sessions happen at the required pace.

This means the system checks:

* how many sessions were completed in the current cadence window

* whether that count meets the required cadence before the window expires

If the cadence requirement is met, the challenge continues.  
If it is not met in time, the challenge fails.

## **When cadence enforcement runs**

Cadence enforcement runs during daily maintenance.

It applies to challenges that are:

* RECURRING

* IN\_PROGRESS

* not currently executing

This means the cadence check is part of the system’s daily rule enforcement rather than something handled manually.

## **Core logic**

Each recurring challenge stores:

* a cadence unit

* a required session count

* a cadence text field, which stores the readable cadence rule for every RECURRING challenge

* a cadence progress counter, which tracks how many sessions the challenge completed in the current cadence window

* a cadence period start timestamp

The backend then determines:

* the length of the current cadence window

* whether that window has expired

* whether the required number of sessions was completed inside that window

## **Cadence Unit / Cadence Windows**
The cadence window depends on the challenge’s configured cadence unit.

Current logic:

* DAILY → 1 day

* WEEKLY → 7 days

* MONTHLY → 30 days

* CUSTOM\_DAYS → the cadence window is derived from the stored cadence text, which is required for all RECURRING challenges  
* For CUSTOM\_DAYS, the backend tries to extract the custom day interval from that stored cadence text

## **What counts as progress**

Progress is tracked through cadenceProgressCounter.

This counter increases when a recurring challenge completes a 21-minute session tick.

So cadence is not based on:

* submission

* promotion

* launch alone

It is based on:

* completed recurring sessions

## **Enforcement outcomes**
When the cadence window expires, the system checks whether the challenge met its requirement.

### **If the requirement is met**
The challenge remains valid.  
The system:

* resets cadenceProgressCounter to 0

* starts a new cadence window from the current maintenance time

### **If the requirement is not met**
The challenge fails.  
The system:

* sets the status to FAILED

* records a failure reason

* updates completion / activity timestamps

## **Why currently executing challenges are excluded**

Challenges that are currently executing are not cadence-checked in that same state.

Purpose:

* avoid failing a challenge while it is actively being played

* ensure cadence is evaluated on completed progress, not mid-session state

## **Why this matters**

Cadence enforcement makes recurring challenges meaningfully different from simple multi-session challenges.

It ensures that:

* RECURRING challenges are judged on whether sessions happen at the required pace, not only on total session count

* a RECURRING challenge can fail before full completion if it falls behind its required pace

## **Relationship with other systems**

This appendix connects to:

* System Architecture → Time & Session System

* System Architecture → Maintenance Logic

* System Architecture → Challenge Lifecycle System

* Features → Scheduling / Execution Slots

Together, these sections explain:

* how recurring challenges are scheduled

* how their sessions are executed

* how cadence windows are enforced over time

* how failure can result from broken pacing, not only from lack of completion

# **Auction & Digout System**
## **Purpose**

This system allows the community to bypass normal challenge progression in specific cases by reviving archived ideas or forcing execution priority through collective support.

---

## **Auction**
An auction is a threshold-based execution mechanic.

### **How it works**
the streamer can assign a challenge a NUMBERS threshold

the community can collectively contribute toward that threshold

if the threshold is reached, the challenge is immediately selected for execution

### **Role in the system**

Auction creates a strategic override on top of the normal ranking system.

**It:**

* encourages collaboration around a shared execution goal  
* creates a high-cost spending path for challenges the community wants strongly enough  
* gives the streamer a flexible override tool for challenges that may seem difficult, marginal, or not worth executing under normal conditions  
* makes community demand more legible by showing which challenges can attract unusually strong collective support

### **Selection logic**

A high threshold can also act as a demand test: if a challenge seems especially hard or not immediately relevant, the community must show stronger support to justify execution.

---

## **Digout**
Digout is the revival mechanic for archived challenges.

### **How it works**

an archived challenge can be revived by paying a percentage of its historical NUMBERS support

once revived, the challenge keeps its previously accumulated total NUMBERS support instead of returning to zero

### **Role in the system**

Digout prevents strong ideas from disappearing permanently after one failed cycle.

**It:**

* restores past support  
* gives previously backed challenges a return path that preserves the value already spent on them  
* prevents the system from treating each challenge cycle as if the past support never happened

---

## **Archive and Revival Logic**
### **Archive conditions**
A challenge moves to Archived when:

* it reaches the end of its 21-day active window without enough promotion

* an auctioned challenge fails to reach its required NUMBERS threshold

### **Digout cost**
Any user can revive an Archived challenge by paying 21% of the total NUMBERS previously spent on that challenge.

### **Support restoration**

A successful digout does not restart the challenge from zero.

The challenge keeps its previously accumulated total NUMBERS support.

### **Why this matters**

Because challenge ranking is based on total NUMBERS spent, restoring historical support allows a revived challenge to return immediately to the competitive pool instead of rebuilding support from scratch.

---

## **Purpose in the product**

Together, Auction and Digout add strategic recovery and override mechanics to the product.

They ensure the challenge system is not only linear, but resilient and dynamic over time.

# **Challenge Removals**
## **Purpose**

Challenge removal allows the original author of a challenge to withdraw it from the active system under defined conditions.

This is not only a cleanup action. It is also an economic and strategic feature that:

* lets authors stop backing ideas they no longer want to keep active  
* creates refund distribution choices  
* allows part of the removed value to be redirected toward the author, the pushers, or the Community Chest

## **Who can use it**

Only the original challenge author can remove a challenge.

This means:

* removals are author-controlled  
* other users cannot remove someone else’s challenge  
* the action is tied to challenge ownership, not general moderation power

## **When removal is allowed**

A challenge can only be removed when it is still in a removable state.

In practice, removal is blocked for challenges that are already:

* Archived  
* Auctioned  
* Completed  
* Failed  
* In Progress  
* Removed

This means removal remains an author-side decision during the active life of a challenge, not something that happens after the challenge has already reached a final state.

## **What removal does**

When a valid removal happens:

* the challenge moves to the Removed state  
* the challenge becomes non-interactable  
* the challenge is terminated as an active progression object  
* a 21% refund amount is calculated from the NUMBERS previously spent on the challenge

The refund is not a full reversal.  
 It is a partial redistribution mechanic based on prior spending.

## **Refund calculation**

When a challenge is removed, the system calculates one refund amount equal to 21% of the NUMBERS previously spent on pushes for that challenge.

That refund amount is then divided into:

* **the author’s share**, based on the author’s own push contribution  
* **the pushers’ share**, based on the contribution made by other pushers

That refund amount is then distributed according to the selected refund option.

## **Refund distribution options**

The author chooses how the calculated refund is distributed.

#### **community\_forfeit**

100% goes to the Community Chest

#### **author\_and\_chest**

the author receives their share; the rest goes to the Community Chest

#### **author\_and\_pushers**

the author receives their share; the rest goes back to pushers

## **Command input**

When removal is triggered through chat command flow, the author selects the refund option as part of the remove command.

This keeps the removal decision:

* explicit at the moment of removal  
* tied to one chosen refund path  
* consistent with the command-driven interaction model

## **Role in the system**

Challenge removal adds a controlled exit path for submitted ideas.

It allows the system to:

* let authors withdraw an active challenge they no longer want to keep in play  
* return part of the previously committed value instead of treating that value as fully lost  
* create strategic choices around whether that returned value benefits the author, the pushers, or the Community Chest  
* encourage stronger follow-through during the active life of a challenge, because both the author and the pushers can have value tied to how the idea performs

This makes challenge authorship more meaningful, because the author keeps agency over an idea after submission, while pushers are also incentivized to track what happens to the challenge over time.

It also makes author commitment more visible, because other users can take the author’s own involvement and removal behavior into account before deciding to support a challenge.

## **Economic role**

Challenge removal is also an economic feature.

It can:

* return part of the value to the author  
* return part of the value to pushers  
* redirect part or all of that value to the Community Chest

This makes removal more than a simple delete action.

It turns withdrawal into a structured redistribution mechanic inside the NUMBERS economy.

## **Product impact**

This feature ensures that:

* challenge authors keep agency over their own proposals  
* abandoned or unwanted ideas can leave the system in a structured way  
* prior support can still create value after removal  
* removal becomes part of the product’s strategic and social economy, not just a destructive action

# **Promotion System (Push)**
## **Purpose**

The Promotion system allows users to express preference and influence which challenges should be prioritized for execution.

It is the primary mechanism through which the community expresses priority and drives outcomes.

---

## **What users can do**

Users can:

* support challenges by spending NUMBERS

* increase a challenge’s priority within the system

* contribute to collective decision-making

---

## **How it works**

Each promotion (“push”) adds more NUMBERS to a challenge’s total support.

* challenges are ranked based on accumulated support, which determines eligibility for execution

* higher-supported challenges gain higher priority within the ranked pool

* eligible challenges then compete for limited execution slots

* multiple users can contribute to the same challenge

This creates a shared prioritization layer across all active challenges.

---

## **Ranking – Eligibility – Execution**
Challenges are first ranked by total NUMBERS support. Only the 21 highest-ranked challenges enter the eligible execution pool.

#### **This creates two levels:**

* ranking (all challenges compete)  
* eligibility (only the 21 highest-ranked challenges can enter the execution pool)

#### **Role in the system**

* This layer filters which challenges can realistically be executed.

#### **It ensures that:**

* execution is not based on visibility alone  
* only sufficiently supported challenges compete for limited execution time

#### **Product impact**

This system ensures that:

* community consensus determines which challenges reach execution  
* low-supported challenges do not compete equally with high-supported ones  
* promotion has a threshold effect, not just a linear effect

---

## **Challenge Selection Process**
Promotion determines rank and eligibility, but it does not guarantee automatic execution.

**This means:**

* challenges first compete for rank through community support

* only eligible challenges move into the eligible execution pool

* final execution selection is still performed by the streamer ("21")

**Final selection can happen:**

* individually by the streamer

* with community input

* through stronger community-driven scheduling over time

This ensures that promotion shapes execution priority without removing final human selection control.

---

## **Cost logic**

The system uses a dynamic cost model:

* each additional push by the same user on the same challenge becomes more expensive

* the rule applies per user and per challenge

This encourages:

* distribution of influence across multiple challenges

* strategic decision-making

* collaboration instead of domination

---

## **Confirmation flow**
Pushes follow a two-stage flow:

* quote generation  
* quote confirmation

This means:

* the system calculates the cost first  
* the user sees the quoted cost before payment  
* the quote remains valid for 30 seconds  
* the push is only applied after confirmation

Purpose in the system:

* makes the quadratic cost visible before commitment  
* reduces accidental spending  
* keeps high-cost promotion actions deliberate

---

## **Role in the system**

The Promotion system transforms user intent into measurable impact.

It creates:

* competition between challenges

* collective prioritization before final execution selection

* a direct connection between user actions, challenge ranking, and limited execution capacity

---

## **Product impact**

This feature ensures that:

* the community actively shapes what gets executed

* users influence outcomes through their pushes, with impact depending on how they distribute their NUMBERS across challenges

* outcomes are driven by the combined actions of multiple users rather than a single individual

# **Slots**
# **Scheduling / Execution Slots**

## **Purpose**

This system defines how limited stream time is turned into executable challenge time.

## **Core principle**

Execution capacity is limited.

Challenges do not move directly from promotion to execution. They must still pass through ranking, selection, and execution-capacity constraints.

They compete for a finite number of execution slots.

For the prioritization side of this process, see the Promotion (Push) section.

---

## **Execution slots**
An execution slot is a predefined 21-minute block reserved for challenge execution.

**These slots:**

* represent the available execution time in a stream.  
* define how much challenge work can actually be performed  
* create a real scheduling constraint inside the product

**Example**

* A stream might be scheduled with 9 execution slots:  
* 9 × 21-minute sessions \= 189 minutes of challenge execution time (3 hours and 9 minutes), excluding breaks or pauses.

### **Scheduling flexibility**

Slot count can vary from one stream session to another depending on available time and stream conditions. Execution slots can be planned by the streamer and may later incorporate stronger community involvement in scheduling.

## **Prioritization logic**

Execution slots define capacity, not priority.

Priority is determined upstream through the Promotion (Push) system.

---

## **Role in the system**

This is where final challenge selection becomes concrete, because eligible challenges must still fit into the limited execution capacity of the stream.

Execution slots connect:

* challenge promotion  
* challenge selection  
* real creator time

They make limited time visible and actionable.

## **Product impact**

This system ensures that:

* stream time is treated as a finite resource  
* promotion has concrete scheduling consequences  
* challenges compete for actual execution capacity, not only visibility

# **Challenge Lifecycle System**
## **Overview**

The Challenge Lifecycle defines how a challenge evolves over time. It defines the states a challenge can move through, from creation to completion, removal, or archival. This system is central to the product because it drives:

* gameplay progression

* user decisions

* economic flow

---

## **Core Principle**

A challenge is not static. It moves through a sequence of states based on:

* time

* user actions

* system logic

---

## **Time & Execution Model**

Challenges evolve over time based on a structured time system.

#### **Session unit: 21 minutes**

base execution unit

#### **Duration types:**
* ONE\_OFF → **must be** **completed** within a **single stream day**, across one or multiple 21-minute sessions

* RECURRING → executed across multiple sessions following a **defined cadence** over time

#### **Cadence (for RECURRING challenges)**
Defines how sessions are scheduled over time (e.g. daily, weekly, monthly, or custom intervals)

Recurring challenges require more than multiple sessions. They also require those sessions to happen at the required pace.

**This means:**

* a recurring challenge defines a cadence window

* the system tracks how many sessions were completed within that window

* if the required count is reached, the cadence window resets

* if the required count is not reached in time, the challenge fails

This makes pacing part of the challenge’s success conditions and distinguishes RECURRING execution from simple ONE\_OFF multi-session completion.

#### **Challenge lifespan: 21 days**
active period before archival if not executed

#### **Commitment window**

The 21-day lifespan starts when a challenge is submitted to the system.

If the challenge does not gain enough promotion within that window, it moves

to Archived.

#### **Stream Day Logic**

A challenge’s active lifespan is measured through Stream Days rather than simple calendar time.

This means:

* one Stream Day is counted when at least one session is executed

* the 21-day active period is measured in Stream Days

* archival depends on Stream Days, not Real Days

This system defines how challenges evolve over time and how execution is distributed across sessions.

---

## **Main States**
A challenge can exist in one of the following states:

* **Active**: The challenge is live and can receive interactions. This includes:

  * pushes

  * disruptions

  * auction contributions (when applicable)

* **In Progress**: The challenge has started execution but has not yet completed all of its required sessions.

  * Important: “In Progress” does not mean the challenge is necessarily being played right now. The real-time execution state is tracked separately through the isExecuting flag.

* **Completed**: All required sessions for the challenge have been completed.

* **Failed**: The challenge started execution but did not meet its execution requirements or cadence rules.

* **Archived**: The challenge is no longer active, is no longer eligible for normal progression, and is kept in system history.

* **Removed**: The challenge was manually removed from the system. This can trigger:

  * refunds

  * redistribution of value

* **Under Review****:** The challenge is in a review-sensitive state before normal lifecycle handling is fully confirmed.

* **Auctioned**: The challenge has been placed into an auction state by the streamer and assigned a NUMBERS threshold.

  * If the community reaches that threshold, the challenge is immediately selected for execution regardless of its normal ranking.

  * If the threshold is not reached, the challenge can move to Archived according to the auction outcome rules.

---

## **State Transitions**

Challenges move between states through:

* **Time**

  * expiration (e.g. 21 Stream Day day limit)

  * execution scheduling

* **User Actions**

  * pushing a challenge

  * removing a challenge

  * reviving a challenge (digout)

  * participating in an auction by contributing toward the NUMBERS threshold

* **System Events**

  * session tick (21-minute session completion)

  * session completion updates

  * daily maintenance (e.g. archival checks)

  * stream state (live / offline) changes affecting execution flow

---

## **State Transition Mental Model**

A challenge follows a simple progression model:

| Under Review → Active → (optional Auctioned) → In Progress → Final State |
| :---- |

In practice, that means:

**Under Review**: The challenge is waiting for validation or decision.

**Active**: The challenge is live in the system and can receive community interaction.

**Auctioned**: The challenge has been assigned a threshold-based execution condition.

**In Progress**: The challenge has started execution but still has remaining sessions to complete.

**Completed / Failed / Removed / Archived**: The challenge exits active progression and becomes part of system history, with the possibility of revival in specific cases. 

This model is important because it separates three distinct layers:

* availability in the system

* execution progress

* final resolution

It also makes clear that:

* a challenge can be active without being executed yet

* a challenge can be in progress without being executed at this exact moment

* a challenge only becomes completed when all required sessions have been successfully completed.

---

## **Economic Impact**

Each state transition can affect the system economy. **Examples**:

* **pushing** → increases challenge value  
* **removal** → triggers partial refunds and redistribution  
* **completion →** ends the challenge as an active progression object

---

## **Lifecycle Importance**

The lifecycle system ensures:

* challenges remain dynamic

* users can influence outcomes

* the system continuously evolves

It transforms challenges from static entries into active game elements.

---

## **Relationship with Other Systems**

This system interacts with:

* backend challenge logic  
* clock / maintenance logic  
* stream state logic  
* NUMBERS and transaction system  
* Game Master control logic

## **Why This Matters**

The Challenge Lifecycle is the core game loop. It defines:

* how users interact with the system

* how value moves

* how the game progresses over time

Without it, the product would be a static database instead of a dynamic system.

# **Maintenance Logic**
## **Overview**

Maintenance is the daily backend process that advances system time and applies rule checks that should not depend on manual intervention.

It is scheduled to run once per day at 21:00 UTC.

## **What it does**

Daily maintenance is responsible for:

* advancing the Real Day clock  
* advancing the global Stream Day clock when a stream occurred during the maintenance window  
* processing user daily tick logic  
* enforcing cadence rules for RECURRING challenges  
* archiving expired challenges when the conditions are met  
* failing unfinished ONE\_OFF challenges when the contiguity rule is broken

---

## **User daily tick logic**
During maintenance, the system updates user activity counters based on what happened since the previous maintenance point.

**This means:**

* users are checked once per Real Day  
* the system determines whether they were active during the maintenance window  
* live activity increments Stream Day-related user counters  
* non-live activity increments offline activity counters

### **Purpose in the system**

This keeps user activity history aligned with the two-clock model and preserves the distinction between live participation and offline participation.

---

## **Safety conditions**
Maintenance does not apply every rule blindly.

Some checks depend on stream conditions.

**This means:**

* cadence checks always run  
* archival and ONE\_OFF contiguity checks only run when a stream occurred during the window  
* those checks are deferred if a stream is currently live

### **Purpose in the system**

This prevents the system from failing or archiving challenges at the wrong moment, especially while a stream is still active.

---

## **Why this matters**

Maintenance is the system’s daily control layer.

**It ensures that:**

* time-based rules are enforced consistently  
* user activity counters remain accurate  
* challenge state changes happen even without manual oversight  
* daily progression stays synchronized with actual stream activity

# **Time & Session System**
## **Time Standard**

The system uses UTC as its reference time standard.

### **Why UTC**

* keeps the product consistent across platforms and countries  
* avoids local timezone ambiguity  
* makes daily resets and time-based rules predictable for a worldwide audience

### **Daily reset point**
* daily resets and daily clock transitions are anchored to 21:00 UTC

---

## Two-Watermark Clock System
The system tracks time using two distinct clocks with different roles.

### **Stream Days (Execution Clock)**
Counts the number of UTC calendar days in which at least one 21-minute session was executed.

Multiple sessions in the same day count as one Stream Day.

**Purpose**:

* measure actual execution effort  
* track content production over time  
* support archival rules and long-term activity tracking

### **Real Days (System Clock)**
Counts the number of calendar days since the project started.

Increments daily at 21:00 UTC.

**Purpose**:

* enforce cadence windows, daily submission resets, and other Real Day-based rules  
* provide the time reference used for recurring challenge cadence checks  
* keep Real Day progression independent from livestream activity

### **Why two clocks**

Execution and progression are not the same.

* Stream Days reflect actual work done  
* Real Days reflect system time passing

**This separation ensures that:**

* challenge archival and other progression rules tied to actual execution depend on real activity, not just time passing.  
* system mechanics remain consistent even without streams

---

## **Daily Reset Logic**

Some system rules are evaluated or refreshed every day at 21:00 UTC.

### **This includes**

* submission quadratic cost scaling  
* daily maintenance-driven user activity tracking  
* other daily maintenance rules tied to system time such as RECURRING cadence enforcement, archival checks, and ONE\_OFF contiguity checks

### **Purpose in the system**

This keeps daily submission cost escalation meaningful while giving users a predictable reset point each day.

---

## **User daily tick logic**
The system updates user activity counters during daily maintenance.

### **This means:**

* users are checked once per Real Day  
* the system determines whether they were active since the last maintenance point  
* live activity increments Stream Day-related user counters  
* non-live activity increments offline activity counters

### **Purpose in the system:**

* keeps user activity history aligned with the two-clock model  
* distinguishes live participation from offline participation over time

---

## **Recurring cadence enforcement**

For RECURRING challenges, the backend checks whether the required number of sessions was completed within the allowed cadence window.

If the requirement is met, the cadence window resets.

If it is not met in time, the challenge fails.

---

## **Stream-day recovery and duplicate-day protection**

The system protects Stream Day logic from being advanced incorrectly.  
**This means:**

* stream state can be recovered after restart

* multiple streams on the same UTC calendar day do not count as multiple Stream Days

## **Identity Records**

The system is structured around three main entities:

### **User**
The central identity in the system. It represents:

* a unique player

* aggregated activity

* global stats

### **Account**
Represents one platform-specific identity record linked to a user.

It includes: 

* platform name

* platform identifier

* username / display name

* platform-specific balance and activity tracking

**Purpose:** 

Store the external identity context used for authentication and user resolution.

### **Perennial Token**
A persistent authentication token stored in the database and managed by the system. **Each token is**:

* linked to a user

* linked to a specific account

* manually enabled or disabled by the system operators

**Purpose**: This enables:

* long-lived access (e.g. trusted users, admin, testing)

* bypassing JWT expiration when needed

* controlled access without re-authentication

  ---

## **How external identity becomes internal identity**
The system resolves an external platform identity into one internal DMG user identity.

**In practice:**

(`platformId` \+ `platformName`) → centralUserId

This is the core identity-resolution rule used before protected actions are processed.

Username and display name remain mutable account context, but they are not the canonical identity key.

---

## **Authentication Methods**
The system supports two authentication methods:

### **JWT Tokens**

**Used for**: standard user sessions, temporary authentication

**Properties:**

* signed using a secret

* time-limited (default: 21 minutes)

* include user and platform identity information

### **Perennial Tokens**
**Used for**: persistent access, internal tools, advanced users 

**Properties:**

* stored in database

* manually activated/deactivated

* no expiration by default

  ---

## **How Identity Is Verified**
When a token is received:

* **Step 1**: Check whether the token exists in the PerennialToken table. If valid, return the associated user and account identity.

* **Step 2**: If not found, verify it as a standard JWT.

This creates a hybrid system:

* flexible for development and power users

* secure for standard usage

### **Token Payload**
Each token contains: userId, platformId, platformName, and username.

**Purpose**: This ensures that every request carries identity context and can be processed without additional lookups.

---

## **Token Lifetime Control**

JWT tokens use controlled expiration.

* **Default**: 21 minutes  
* **Validation**: duration is sanitized and capped at a maximum defined in config.

**Purpose**: Prevents overly long sessions and unsafe token usage.

---

## **Identity in the Frontend**
The frontend uses tokens to:

* verify access to protected pages  
* display user identity (username \+ platform name)  
* personalize interactions

**Example**: Explorer and Submit Form require token verification before access and display user identity through the header component.

---

## **Link and Identity Match**

### **Purpose**

This logic ensures that protected web actions are performed by the same user who initiated the request from chat.

### **How it works**

* chat platforms remain the primary identity source  
* when a user requests access to a protected web feature from chat, the system generates a user-specific access link  
* the web action is then verified against the originating identity before any protected action is allowed

### **Why it matters**

* improves security by tying protected actions to the same verified identity that requested the link  
* keeps the access flow fast and usable by avoiding separate website accounts and passwords  
* prevents NUMBERS spending from being applied to the wrong platform account

### **Important note**

Private links should be treated like temporary login links, not like harmless share links. Anyone holding the link may be able to act with the associated identity until the link expires or is revoked.

---

## **Access Control**

The token system enables fine control over access:

* **Public endpoints**: no token required  
* **Protected endpoints**: require valid token  
* **Admin endpoints**: require elevated identity (e.g. Game Master)  
  ---

## **Security Model**

The system ensures:

* signed token verification

* expiration control

* database-stored persistent tokens

* separation between public and protected routes

  ---

## **Current Strengths**

* supports multi-platform identity

* flexible authentication (JWT \+ persistent tokens)

* clean separation between identity and UI

* scalable for future access control systems

  ---

## **Planned Evolution**

The system is designed to support future features:

* Paid access system: determine whether a user can enter a protected feature, what level of access they have, and how long that access remains valid.

* Session-based permissions: allow temporary elevated access and feature unlocking.

* Stronger identity linking logic: improve cross-platform merging and user-level progression.

  ---

## **Why This System Matters**

This system is critical because it:

* connects all user actions to a verified identity

* enables controlled access to features

* supports monetization and scaling

It acts as the backbone of **security**, **personalization**, and **game integrity**.

# **Backend Logic**
## **Purpose**

The backend logic layer applies the rules of the product.

It is responsible for turning stored data and incoming actions into actual system behavior.

This includes:

* challenge lifecycle enforcement  
* cost calculation  
* stream-state-dependent logic  
* clock progression  
* access validation  
* synchronization with external systems

## **Core principle**

The backend applies the core rules of the product in dedicated service layers rather than inside frontend pages.

This means:

* the main product rules are applied in backend services  
* time-based behavior is handled in dedicated backend logic  
* actions that affect multiple records are coordinated in backend logic  
* route handlers translate known service failures into consistent HTTP responses  
* the system behaves consistently across chat, web pages, and admin flows

---

## **Main logic areas**

### **Challenge logic**
Handled primarily in `challengeService.ts`.

This logic controls:

* challenge submission  
* push quote and push confirmation  
* digout  
* removals  
* disruption placeholder logic  
* execution launch  
* 21-minute session progression and completion checks  
* challenge archival  
* status changes

Why it matters:

* this is where the core challenge mechanics are enforced  
* it connects challenge lifecycle, NUMBERS spending, stream metrics, and event dispatching

### **Autonomous session scheduler**
Session progression is checked by an autonomous scheduler rather than being tracked manually.

This works as follows:

* the scheduler checks execution state at a fixed interval

* it verifies whether the current session has reached the 21-minute threshold

* once the threshold is reached, it increments the session count and stops active execution

* if the final required session is reached, the challenge is marked as completed

* additional sessions require manual re-execution by the Game Master

* actual live execution is tracked by the isExecuting flag, which is separate from the broader lifecycle state In Progress

Why it matters:

* it makes session progression time-based rather than manually tracked  
* it enforces the 21-minute execution unit consistently  
* it keeps execution stopping and completion logic reliable across sessions

### **Clock and maintenance logic**

Handled primarily in `clockService.ts`.

This logic controls:

* daily maintenance  
* user daily tick processing  
* ONE\_OFF contiguity enforcement  
* recurring cadence enforcement  
* archival checks tied to maintenance conditions

Why it matters:

* it applies the product’s time rules centrally  
* it keeps challenge progression, user counters, and maintenance behavior synchronized

### **Stream-state logic**

Handled primarily in `streamService.ts`.

This logic controls:

* live / offline transitions  
* stream session creation and finalization  
* global Stream Day advancement  
* per-challenge Stream Day advancement for active challenges  
* in-memory stream-state recovery after restart

Why it matters:

* stream state affects challenge launch context, live pricing rules, Stream Day advancement, and active-challenge day progression

* this logic ensures Stream Days only advance when real stream activity happens

* it also allows the backend to restore its live-stream state after restart by loading the current active session back into memory from the database

### **Token and identity logic**

Handled primarily in `jwtService.ts` and related token routes.

This logic controls:

* JWT generation  
* token verification  
* perennial token lookup  
* duration validation and normalization  
* identity payload handling

Why it matters:

* it ensures protected actions are tied to verified identity  
* it supports both temporary and persistent access flows

### **Authentication and request identity resolution**
Handled primarily in `authMiddleware.ts` together with `findOrCreateUser()` in `userService.ts`.

This logic controls:

* reading the external platform identity from the incoming command payload  
* resolving the matching internal DMG user from platformName \+ platformId  
* creating a new User / Account pair when the identity does not yet exist  
* attaching the resolved internal DMG user identity and request context before command execution  
* enforcing Game Master\-only access on privileged routes

Why it matters:

* it turns external chat-platform identity into authenticated internal DMG identity before backend logic runs  
* it allows first-time users to enter the system simply by sending a command  
* it keeps command execution tied to a resolved internal user rather than raw external payload alone  
* it makes the chat command layer the practical runtime entrypoint into the product

### **Lumia balance synchronization**
Handled primarily through `lumiaService.ts` together with challenge-related service logic.

This logic controls:

* authoritative NUMBERS deductions and additions  
* synchronization between external platform balances and local database state  
* balance updates after paid actions succeed

Why it matters:

* Lumia acts as the external source of truth for platform balances  
* the backend must keep local records aligned with that external balance state  
* paid actions depend on this synchronization to remain reliable

### **Transaction and payment synchronization logic**

Handled across service functions that combine:

* external balance calls

* database updates

* temporary pre-confirmation records

* deletion of temporary quote records after successful confirmation

This logic controls:

* optimistic local balance pre-checks based on the latest balance stored in the DMG database, before calling the Lumia API

* paid-action execution after balance confirmation

* multi-record database updates after payment success

* cleanup of temporary push quote records

Why it matters:

* optimistic local balance pre-checks reduce unnecessary external balance calls and stop obviously invalid actions early before full transaction execution

* it keeps local state and external balances aligned

* it uses Prisma transactions, so paid actions either complete fully or fail without leaving partial database updates

### **User activity and identity resolution**

Handled primarily in `userService.ts`.

This logic controls:

* user creation and account creation  
* platform-account lookup  
* user activity recording  
* Game Master identity checks  
* user statistics aggregation

Why it matters:

* it connects external platform identity to internal DMG identity  
* it keeps user activity and account linking consistent across the system

### **Event dispatching**
Handled in `eventService.ts`.

This logic publishes challenge-related events so other parts of the system can react to meaningful state changes such as:

* submission  
* execution  
* session completion  
* digout  
* removal  
* GM activation

Why it matters:

* it separates core state changes from downstream reactions  
* it allows notifications, console subscribers, and future integrations to react without being hard-wired into the core service logic  
* it prepares the system for future subscribers without coupling them directly to challengeService or scheduler logic  
* subscribers

### **Rate limiting and infrastructure support**
Handled in `upstashService.ts`.

This logic provides Redis-backed rate limiting for command flows and helps protect the system from excessive repeated requests.

Why it matters:

* it reduces abuse risk  
* it protects command-driven interaction during live usage

### **Command dispatch logic**
Handled primarily in `commandDispatcher.ts`.

This logic controls:

* receiving normalized command payloads from the dedicated command route  
* parsing command arguments  
* routing each command to the correct service function  
* keeping command handling centralized behind one backend entry point

Why it matters:

* it gives the system one consistent command-processing layer  
* it makes authentication and rate limiting easier to apply before command execution  
* it reduces duplication across command sources that use the same backend entry point

### **Reference metadata resolution**

Handled primarily in `referenceMetadata.ts`.

This logic controls:

* validating reference URLs  
* resolving reference titles from supported public sources  
* improving the challenge submission flow by reducing manual entry

Supported sources currently include:

* YouTube  
* Vimeo  
* SoundCloud  
* Spotify  
* Wikipedia  
* generic public pages

Why it matters:

* it improves submission UX without changing the core game logic  
* it makes reference-based challenges easier to create and review  
* it keeps title-fetching behavior centralized in one helper layer

---

## **Important implementation choices**

### **UTC-only time handling**

Time-sensitive logic uses UTC as the reference standard.

This affects:

* daily resets  
* maintenance  
* Real Day progression  
* Stream Day logic  
* quote expiration  
* consistent timestamps across multi-step service operations

Why it matters:

* it prevents timezone drift and daylight-saving ambiguity  
* it gives the system one consistent time standard for all clocks and resets

### **Canonical challenge activity timestamp**

Each challenge keeps one canonical last-activity timestamp.

This timestamp is refreshed whenever a meaningful change happens to the challenge, such as:

* submission  
* push  
* execution  
* completion  
* archival  
* removal  
* other state-changing actions

Why it matters:

* it gives the system one reliable reference point for recency  
* it supports synchronization, ordering, monitoring, and incremental frontend updates

### **Precision timing rules**

Different timing needs use different time precision levels.

This includes:

* standardized UTC timestamps for multi-step service operations  
* millisecond-level timing for short expiration windows such as push quotes

Why it matters:

* it keeps multi-record updates consistent inside one operation  
* it makes short-lived expiration logic reliable where higher precision is required

### **Two-clock logic**
The backend treats Real Days and Stream Days as different clocks.

This means:

* Real Days advance through maintenance  
* Stream Days advance only when a qualifying stream day occurs  
* challenge progression and archival can depend on actual stream activity rather than passive time alone

Why it matters:

* it preserves the product’s distinction between system time and execution effort

### **Just-in-time daily reset behavior**

Some daily user resets are not applied through a full database-wide midnight reset.

Instead, they are checked and refreshed when the user next interacts after the reset boundary.

This currently affects:

* daily submission cost scaling

Why it matters:

* it reduces unnecessary background load  
* it keeps daily reset behavior predictable without forcing full-system rewrites at reset time

### Two-stage push flow
Push execution is split into two stages:

1. quote generation  
2. quote confirmation

This means:

* the cost is calculated first  
* a temporary quote is stored briefly  
* the final push is only committed after confirmation

Why it matters:

* it makes quadratic push cost explicit before payment  
* it supports controlled confirmation and atomic execution

### **BigInt financial handling**
Large financial ledgers use `BigInt`.

This is used for:

* game-wide totals  
* large accumulated NUMBERS metrics  
* community chest and other system-wide financial totals  
* other long-term aggregate financial counters

Because JSON does not serialize BigInt values by default, the backend includes explicit serialization handling before returning these values to the frontend.

Why it matters:

* it prevents overflow risk in large financial ledgers  
* it keeps long-term economy tracking safe as the system grows

### **Central rule configuration**

Core rule values are centralized in `gameConfig.ts`.

This includes:

* base costs  
* timing constants  
* percentage rules  
* token duration limits  
* admin/system identifiers  
* precision helpers used in BigInt calculations

Why it matters:

* it keeps core game rules explicit and editable in one place  
* it reduces hard-coded rule duplication across services  
* it makes balancing and system adjustment easier over time

---

## **Relationship with other technical layers**

This backend logic sits between:

* the database layer  
* the API route layer  
* external integrations such as Lumia  
* frontend interfaces that trigger system actions

It is the rule-enforcement layer of the product.

## **Why this backend logic matters**

This is where DMG becomes an actual system rather than a static website or database.

It ensures that:

* product rules are enforceable  
* actions have consistent consequences  
* time, economy, identity, and execution stay synchronized  
* live interaction can scale without relying on manual control

# **Database**
## **Purpose**

The database stores the persistent state of the product.  
 It holds the core records needed to track:

* user identity  
* platform accounts  
* challenges  
* pushes  
* stream sessions  
* maintenance clocks  
* persistent access tokens

It is the durable system layer that allows DMG to preserve history, apply rules consistently, and recover state across sessions.

## **Core design principle**

The database is structured around a few key separations:

* **internal user identity** is separated from **external platform accounts**  
* **challenge state** is separated from **push** **history**  
* **global time** **tracking** is separated from **individual stream sessions**  
* **temporary quote records** are separated from **finalized transactions**

This keeps identity, challenge state, time tracking, and temporary pre-confirmation records clearly separated.

---

## **Main models**

### **User**
Represents the central internal identity of a player in DMG.

It stores:

* the internal user ID  
* game-wide participation counters  
* long-term activity history  
* aggregate NUMBERS metrics  
* progression-related state

Why it exists:

* unify activity across platforms  
* keep global user history in one place  
* support user-level totals, progression, and long-term activity tracking

### **Special system record**
User ID 1 is used as the **global ledger record** (“World User”).

It is used to track:

* game-wide NUMBERS totals  
* Community Chest totals  
* system-wide economy counters tied to spending, refunds, and redistribution

Why it matters:

* it gives the system one central record for global economy tracking  
* it keeps player-level activity separate from system-level financial totals

### **Tracked user metrics**

The `User` model also stores long-term per-user game metrics such as:

* total digouts executed  
* total pushes executed  
* total disrupts executed  
* total challenges submitted  
* total removals executed  
* total NUMBERS spent  
* total value received from removals  
* total value caused by removals  
* total value directed to the Community Chest  
* total value returned to pushers

Why it matters:

* these metrics make user activity, contribution, and economic impact visible over time  
* they support progression, analytics, and future user-facing history features

### **Account**
Represents one platform-specific identity linked to a user.

It stores:

* platform name  
* platform ID  
* mutable username / display name  
* platform-specific NUMBERS balance  
* platform-level activity timestamps

Why it exists:

* separate external identity from internal identity  
* track balances and activity per platform  
* support cross-platform identity linking

### **StreamStat**

Represents the global system clocks and maintenance watermark.

It stores:

* Real Days  
* Stream Days  
* last maintenance timestamp

Why it exists:

* support the two-watermark clock system  
* store the reference point used by daily maintenance  
* keep global time-based mechanics consistent

### **Challenge**

Represents the canonical record of a challenge.

It stores:

* challenge content  
* proposer identity snapshots  
* lifecycle state  
* duration type  
* cadence state  
* execution progress  
* timestamps  
* challenge-level economy metrics

Why it exists:

* act as the central state record for the challenge lifecycle  
* support ranking, execution, archival, and history

### **Push**

Represents finalized promotion history for a challenge.

It stores:

* which user pushed  
* which challenge was pushed  
* cost paid  
* quantity  
* timestamp

Why it exists:

* preserve push history as a permanent record  
* support challenge ranking and user-level push tracking  
* store each finalized push as its own record instead of only keeping a challenge-level total

### **TempQuote**
Represents a temporary push quote before confirmation.

It stores:

* quoted push cost  
* target challenge  
* target user  
* requested quantity  
* creation timestamp  
* lock state

Why it exists:

* support the two-stage push flow  
* separate price calculation from final execution  
* allow short-lived quote validation before committing the action

### **Stream**
Represents one stream session.

It stores:

* session start and end timestamps  
* duration  
* per-session activity totals  
* session-level economy metrics  
* processing state

Why it exists:

* preserve stream session history  
* track per-session activity and economy metrics  
* distinguish an active stream session from a fully finalized one

### **PerennialToken**
Represents a persistent authentication token stored in the database.

It stores:

* token value  
* linked user  
* linked account  
* active / inactive state  
* creation timestamp

Why it exists:

* support persistent access flows beyond short JWT sessions  
* allow manually managed token-based access  
* prepare the system for more flexible entitlement and feature-access models

## **Why this database structure matters**

This structure keeps DMG understandable at the data level.

It allows the system to:

* separate identity, activity, economy, and time clearly  
* preserve history without mixing temporary and permanent state  
* support cross-platform continuity  
* apply system rules on top of clearly separated records for identity, challenges, time, and transactions

# **Integrations**
## **Purpose**

This section documents the external systems DMG relies on to operate across live platforms, community channels, and web tools.

These integrations are used to:

* connect DMG to live chat environments  
* reuse existing loyalty and balance infrastructure at launch  
* extend participation across live and offline channels

They are not the core product logic themselves, but they make the product usable in real conditions.

---

## **Lumia**
### **Purpose**

Lumia is used as the launch-layer bridge between DMG and supported livestream platforms.

### **What it does**

Lumia currently provides two critical capabilities:

* an existing loyalty-points system used as the initial NUMBERS layer  
* a shared command-listening layer across Lumia-supported livestream platforms

This means Lumia can:

* grant points through watch time  
* apply reward modifiers based on user status such as follower, subscriber tier, VIP, or moderator where the connected platform and Lumia support it  
* listen for supported chat commands  
* send command payloads with user identity data to the main command ingress route

### **Role in the system**

Lumia reduces the amount of custom infrastructure DMG needs at launch.

It allows DMG to:

* start with an already working points system instead of building one from scratch  
* reuse an existing live chat-listening layer across supported platforms  
* connect paid actions to an external balance authority rather than treating the local database as the final balance source

### **Why it matters**

This integration is critical because it makes the product operational much faster.

It allows DMG to launch with:

* watch-time-based NUMBERS earning  
* platform-linked reward modifiers  
* supported chat commands routed to the main command ingress route

It also leaves room for future evolution if DMG later moves more of that logic in-house.

---

## **Discord**

### **Purpose**

Discord is the main offline community hub for DMG.

### **What it does**

At baseline, Discord provides:

* a persistent community space outside live sessions  
* continuity between streams  
* a place where ideas, discussion, and coordination can continue when the stream is offline

Its product value for DMG comes mainly from the planned Discord bot layer.

### **Planned Discord bot layer**
A dedicated Discord bot is planned as the main offline interaction layer.

Its role would be to:

* authenticate users against their livestream-platform identity

* listen for supported offline commands

* send valid command payloads to the main command ingress route

* restrict or disable certain offline commands when the stream is live, so live participation remains concentrated on livestream platforms

### **Role in the system**

Discord gives DMG a persistent hub, while the planned bot would turn that hub into an actual offline gameplay layer.

Together, they would allow DMG to:

* stay socially active between streams

* support structured interaction outside live sessions

* reuse the same command model across live and offline channels

### **Why it matters**

Without the bot, Discord is mainly a community hub; with it, Discord becomes part of the DMG interaction model.

That makes offline participation operational rather than only social.

---

## **Streaming platforms**

### **Purpose**

Streaming platforms are the live participation surfaces where DMG runs in real time.

### **What they do**

Streaming platforms provide:

* live chat environments  
* platform-level user identity  
* the real-time context where commands and participation happen during a stream

Depending on the platform and the Lumia integration, they can also support:

* watch-time-based reward logic  
* subscription-aware reward modifiers  
* live participation tied directly to stream state

### **Role in the system**

Streaming platforms are where DMG becomes a live product.

They provide:

* the real-time participation layer  
* the original source of many commands and user actions  
* the live context that drives stream-state-dependent mechanics such as discounts, execution flow, and activity tracking

### **Why it matters**

These integrations allow DMG to function as a multi-platform livestream product rather than a single-site system.

They are the live surfaces on which the product is actually experienced.

---

## **Integration model**

These integrations work together as one operational stack.

At a high level:

* streaming platforms provide the live participation surfaces  
* Lumia provides the launch-layer loyalty and command bridge across supported livestream platforms  
* Discord provides the main offline hub and the future offline bot layer

This allows DMG to operate:

* during live streams  
* between live streams  
* across multiple participation channels without reducing everything to one interface

## **Why this integration layer matters**

The integration layer connects backend rules to real user environments.

It allows DMG to:

* launch on top of existing live infrastructure  
* preserve continuity across live and offline participation  
* support a cross-platform experience while keeping one central system logic

# **Transaction Handling**
## **Purpose**

This section explains how DMG handles multi-step operations that affect both local database state and external balance state.

It focuses on:

* Prisma transactions  
* external balance synchronization  
* rollback behavior  
* failure handling

## **Why this matters**

Many important DMG actions do not update only one record.

They can affect:

* user metrics  
* account balances  
* challenge state  
* stream metrics  
* temporary quote records  
* global economy totals

Because of this, transaction handling is critical to system reliability.

## **Prisma transactions**

DMG uses Prisma transactions for actions that update multiple database records as part of one operation.

This is used for flows such as:

* challenge submission  
* push confirmation  
* digout  
* removals  
* disruption placeholder spending  
* execution state changes  
* automatic session tick updates

### **Purpose in the system**

Prisma transactions ensure that related database updates succeed together or fail together.

This prevents partial local writes such as:

* a challenge submission completing without the related user, account, stream, and global-ledger updates  
* a push confirmation updating challenge totals without updating the matching user, account, stream, and quote records  
* a digout or removal updating challenge state without the matching economy and activity records

## **External balance synchronization**

Some paid actions also depend on Lumia, which acts as the external balance authority.

This means the system often combines:

* a local optimistic balance check  
* an external Lumia deduction or addition  
* local database updates after the external call succeeds

### **Purpose in the system**

This keeps DMG’s local records aligned with the external balance source rather than treating the local database as the final authority for platform balances.

## **Rollback behavior**

When a database transaction fails, the local database changes inside that transaction are not committed.

This means the system avoids partial local outcomes such as:

* challenge state updated without the matching user update  
* stream metrics updated without the matching account update  
* quote confirmation applied without the full set of related writes

### **Important distinction**

Rollback protection applies to local database operations.

External calls are outside the database transaction itself.

Because of this, the system is designed to call external balance operations at controlled points in the flow before final local commit logic is completed.

## **Failure handling**
The backend handles failures explicitly at both the service layer and the route layer.

This includes:

* checking required inputs and action conditions before costly or multi-step operations run  
* rejecting clearly invalid actions before external calls or multi-record updates begin  
* mapping known service errors to clear HTTP responses  
* cleaning up temporary records when needed  
* preventing partially completed local state changes

### **Examples**

* an expired push quote is rejected and removed  
* an insufficient local balance check stops the action before the external call  
* an external Lumia failure stops the paid action before local completion  
* a failed database transaction prevents partial local writes

## **Relationship with Backend Logic**

This section complements Technical Implementation → Backend Logic.

### **Backend Logic explains:**

* what rules the system applies

### **Transaction Handling explains:**

* how the system keeps multi-step operations coherent when those rules are executed
